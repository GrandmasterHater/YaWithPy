Эталонное решение реализует задачу через скрытое внутреннее состояние. Тип робота содержит все операции присуще роботу, но сам робот не хранит состояние, оно пересоздаётся в 
результате каждого вызова операции над роботом. Моё решение отличается от эталонного. Я определил несколько абстракций для движущейся системы и чистящей системы. Тип робота 
реализует операции каждого типа, и хранит текущее состояние. При вызове операции над роботом создаётся новый экземпляр робота в изменённом состоянии. Таким образом сущность
робота неизменна, а тип каждого робота сам определяет доступные для него операции и необходимый набор параметров состояния. Управление роботом реализовано через команды.
По задумке каждая команда контролирует тип с которым работает, так команда Move ожидает, что тип робота будет Movable. Но поскольку я не создал общую иерархию типов робота, 
а лишь разделил по поведению идея реализована не полностью. В питон это будет работать, но типы не будут отслеживаться линтером несмотря на аннотации, покольку исполнение
команд идёт через общий список без указания конкретного типа команд.
